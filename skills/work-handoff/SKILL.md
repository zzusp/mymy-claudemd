# 生成精简工作交接

把交接写成“当前任务快照”，不是代码审计、架构说明书或新会话调查清单。默认控制在约 600–1200 个中文字；除非用户明确要求详细版，否则宁缺毋滥。

## 取舍标准

只写至少满足一项的信息：

1. 新会话仅靠读取当前代码很难快速还原；
2. 会直接改变下一步动作或验收顺序；
3. 遗漏后可能误改、误推、误部署、覆盖脏文件或重复踩坑。

必须保留：

- 当前目标、已生效的范围变化和所处阶段；
- 已完成、进行中、明确未验证，以及唯一的下一步；
- 唯一续办工作目录/worktree、分支、当前提交、脏文件边界，以及本任务全部会影响续办判断的关联 PR/MR；PR/MR 可能有多个，需标明各自仓库、角色、target/base、source/head 和生成时状态；
- 关键设计决策、不可破坏的约束、真实阻塞和高风险注意事项；
- 本任务实际引入的依赖/工具链变化；没有变化时只写一句“依赖无变化”；
- 已完成的关键验证及仍需补的验证。

默认删除：

- 详细代码架构、完整数据流、接口/DTO/文件清单；只有某个非显而易见的架构边界会影响下一步时，才用一条短句保留；
- 全量命令输出、日志、证据索引、提交历史和需求时间线；
- 新会话可从项目指令文件、README、源码或最新远端状态重新获得的内容；
- 容易过期的远端、CI、环境和 PR/MR 检查细节。PR/MR 只保留链接、角色、分支关系和生成时状态，并要求新会话先刷新；
- 通用开发建议、探索步骤、备选方案和没有明确完成判据的 Todo；
- 空章节、重复结论和为了填模板而写的“无/不适用”。

## 生成步骤

1. 以用户指定范围为准；未指定时使用当前会话正在推进的任务，不混入其他历史任务。
2. 读取适用的项目指令和当前任务已有计划/验收文件，只做必要回读。
3. 最小核验当前续办入口：工作目录、分支、提交、脏状态；从任务文档、已知链接和相关分支回查本任务全部关联 PR/MR，不得只记录当前分支自动找到的一个。保留仍影响默认分支、集成分支、发布或后续修订判断的已合并/开放 PR/MR，排除无关历史评审。不要为写交接全面探索代码架构或环境现场。
4. 依赖变化只比较本任务实际改动涉及的 manifest、lockfile 或工具链配置。
5. 按 [精简模板](references/handoff-template.md) 写入项目已有交接/文档目录；无明确且安全的落盘位置时，先在回复中给出内容。
6. 脱敏凭据、账号、PII 和私有连接信息。写完后独立回读，确认文件非空、标题齐全且没有未经证实的“已完成”。

## 新鲜度规则

- 分支、HEAD、脏状态和 PR/MR 都是生成时快照，必须标时间。
- 新会话开始后先重新读取项目指令并刷新这些易变状态，再实施 Todo。
- 不因交接快照与新现场不同而强行恢复旧状态；以新会话核验到的当前事实为准。
